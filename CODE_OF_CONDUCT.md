# Code of Conduct

## Purpose

This project aims to be technically rigorous, direct, and respectful. We want discussion to stay useful for contributors, users, reviewers, and maintainers, even when there is disagreement about design, correctness, or release quality.

## Expected behavior

Participants in this project are expected to:

- Be respectful in language and tone
- Focus criticism on code, behavior, evidence, and technical tradeoffs
- Assume good faith while still being precise about problems
- Accept feedback and correction without escalating into personal attacks
- Keep discussions relevant to the repository and the issue at hand
- Respect differing levels of experience, language fluency, and background

## Unacceptable behavior

The following behavior is not acceptable:

- Harassment, threats, intimidation, or deliberate humiliation
- Personal insults, slurs, or discriminatory language
- Repeated hostility, baiting, or bad-faith argumentation
- Publishing private information without explicit permission
- Sexualized language or unwelcome sexual attention
- Spam, trolling, or intentionally disruptive behavior
- Retaliation against people who report problems or enforce this policy

## Scope

This Code of Conduct applies to:

- GitHub issues, pull requests, reviews, and discussions
- Commit messages and project documentation
- Other public or private project spaces used for collaboration about this repository

## Reporting

If you experience or witness unacceptable behavior, contact the maintainer privately:

- GitHub private contact
- Email: `Vanderhell@gmail.com`

When reporting, include relevant links, screenshots, or context if available.

## Enforcement

The maintainer may take any action considered appropriate, including:

- Asking for a change in behavior
- Editing, hiding, locking, or removing content
- Rejecting contributions
- Temporarily or permanently blocking further participation

Enforcement decisions will be based on protecting the project and its contributors, not on winning arguments.

## Project standards

Strong technical disagreement is allowed and expected. Blunt feedback is acceptable when it remains evidence-based, relevant, and non-abusive. "Direct" is not a license for hostility.
