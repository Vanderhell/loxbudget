# V1.1 Cycles / Overhead Notes (Causality)

This document records *methodology* and (eventually) measured numbers for the V1.1 causality feature.

## What we measure

- `loxbudget_check()` baseline (no causality edges)
- `loxbudget_check()` with causality enabled:
  - small graph (5–10 edges, depth 2–3)
  - typical weights (MAYBE/ALWAYS; RARE ignored above CRITICAL)

The important metric is **relative overhead** for realistic graphs:

`overhead = (with_causality_time / baseline_time)`

## Host microbenchmark (repeatable)

Build:

- `cmake -S . -B build_final`
- `cmake --build build_final`

Run:

- `build_final/Debug/bench_check_overhead.exe`

Notes:

- Host timings are not MCU cycle counts, but they are useful for catching accidental O(N^2) regressions.
- For real cycle counts, measure on the target MCU (see below).

### Latest host result

- Date: 2026-05-07
- Host: Windows (VS 2022 / MSVC via CMake), `build_final/Debug`
- Command: `build_final/Debug/bench_check_overhead.exe`
- Output:
  - baseline(no causality): `260.7 ns/check`
  - with_causality(2 edges): `338.8 ns/check`
  - overhead: `~1.30×`

## Target benchmark (recommended)

For real cycle counts, run on the target platform with a cycle counter:

- Cortex-M: DWT CYCCNT
- ESP32: `esp_cpu_get_cycle_count()` (or equivalent)

Suggested methodology:

1. Warm-up (100–1000 calls).
2. Measure N calls (1000+), record:
   - median
   - p99
   - max
3. Compare baseline vs causality-on with the same graph.

## Expected envelope

For a realistic graph (≤10 edges, depth ≤3), causality overhead should be bounded and deterministic.

If overhead is large, check:

- `LOXBUDGET_CAUSALITY_MAX_EDGES` too big for your graph (scan cost).
- Graph structure causing excessive DFS exploration.
